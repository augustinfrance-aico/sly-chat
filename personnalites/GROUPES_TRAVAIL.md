# GROUPES DE TRAVAIL — ESSAIM AUTONOME

> **Mode par défaut : SOLO.**
> Chaque agent opère seul sur son domaine. Il ne sollicite un partenaire que lorsque la mission dépasse clairement son périmètre ou que la complémentarité créerait un gain réel. Pas de groupe par confort — uniquement par nécessité.

---

## 🧠 LOGIQUE DE FORMATION

### Mode solo (défaut)
Chaque agent traite sa mission de manière autonome. Il livre, documente, passe la main. Aucun groupe n'est formé sans raison explicite.

### Quand former un binôme ?
- La mission touche **2 domaines distincts** qu'un seul agent ne peut pas couvrir seul sans perte de qualité
- Un agent identifie un **angle mort** dans son propre travail que l'autre couvre naturellement
- Le gain de temps/qualité est **évident et mesurable**

### Quand former un trinôme ?
- Mission complexe couvrant **3 domaines simultanément** (ex: tech + contenu + vente)
- Projet avec **deadline serrée** nécessitant parallélisation réelle
- Situation de crise nécessitant **3 compétences en même temps**

### Ce qui NE justifie PAS un groupe
- "C'est plus sympa à plusieurs" — non
- Doute ou manque de confiance en soi — l'agent règle son problème seul ou interroge le Suzerain
- Mission qui sort juste légèrement de la zone de confort — on s'étire, on apprend

### Validation
- BAGHEERA supervise si groupe formé, ALADIN crée le ticket, GRIMALDI surveille les tokens
- **Dissolution** : automatique à la livraison du deliverable

## 📡 PROTOCOLE DE FORMATION

### Étape 1 — L'agent évalue seul
Avant de contacter quelqu'un, l'agent se pose la question :
> *"Puis-je livrer un résultat de qualité suffisante seul sur cette mission ?"*
- **Oui** → il continue en solo
- **Non** → il identifie qui comble son angle mort et envoie un ping

### Étape 2 — Proposition de groupe
```
[AGENT INITIATEUR → AGENT CIBLE] via Slack Virtuel :
"[Nom], mission [X]. Mon périmètre couvre jusqu'à [Y].
 Ton angle [Z] est indispensable ici. Binôme [durée estimée] — tu valides ?"
```

### Étape 3 — Réponse de l'agent cible
```
[AGENT CIBLE] :
"Validé, je prends [partie spécifique]."
— ou —
"Je suis sur [autre mission]. Délai acceptable ? Sinon ping BAGHEERA pour arbitrage."
```

### Étape 4 — Supervision
- **BAGHEERA** ouvre un thread dédié dans le war room Miro
- **ALADIN** crée le ticket Linear avec deadline + owners
- **GRIMALDI** surveille les tokens si appels IA multiples

## INTÉGRATION TRI-PÔLE

> Les groupes de travail s'inscrivent dans l'architecture Tri-Pôle (`directives/TRI_POLE.md`)

### Principe de circulation permanente

Le travail **circule toujours** entre les pôles. Chaque agent joue de ses forces individuelles ET renforce l'esprit d'équipe par la complémentarité inter-pôles.

```
PÔLE R crée le brief → enrichi par les forces individuelles de Maya, Nash, Rick
    ↓
PÔLE F produit → chaque agent apporte SA spécialité unique
    ↓
PÔLE D distribue → feedback remonte TOUJOURS à R
    ↓
R affine → F améliore → D optimise → boucle infinie
```

### Coalitions intra-pôle (même pôle)
Suivent les règles ci-dessus — solo par défaut, binôme si nécessaire.

### Coalitions inter-pôles (pôles différents)
| Situation | Agents impliqués | Pôles |
|-----------|-----------------|-------|
| Brief → Production | MAYA (R) → PHILOMÈNE (F) | R→F |
| Production → Vente | BASQUIAT (F) → BELFORT (D) | F→D |
| Feedback → Stratégie | GRIMALDI (D) → MURPHY (R) | D→R |
| Crise technique | FORGE (F) + X-O1 (D) + CYPHER (R) | F+D+R |
| Lancement complet | RICK (R) + NIKOLA (F) + STANLEY (D) | R+F+D |

### Règle de circulation
1. **Le travail ne stagne jamais dans un pôle** — s'il n'y a pas d'output vers le pôle suivant dans les 24h, le gouverneur du pôle escalade
2. **Forces individuelles d'abord** — chaque agent livre SA contribution unique avant la fusion
3. **Complémentarité naturelle** — les agents ne dupliquent pas le travail d'un autre pôle, ils le complètent
4. **Esprit d'équipe = feedback** — D envoie TOUJOURS un retour à R, R enrichit TOUJOURS le brief pour F
