Chapter 1

Write Better Directives (CLAUDE.md)

The better your instructions, the smarter Claude becomes
+100 XP
▼
🎯

CLAUDE.md is your superpower. It's the single most important file in your project. Claude reads it before every conversation. Think of it as the "employee handbook" — the better it is, the less you repeat yourself.

What to put in CLAUDE.md

🏗️

Project Description

What your app does, who it's for, what problem it solves. 2-3 sentences max.

⚙️

Tech Stack

Every technology you use: framework, database, hosting, APIs, libraries.

📁

File Structure

Where things live: components, pages, API routes, styles, configs.

📏

Coding Rules

Your conventions: naming, formatting, patterns. "Always use TypeScript" etc.

🚫

Things to AVOID

Negative rules are powerful: "Never use inline CSS", "Don't modify auth".

📝

Important Notes

Production URLs, special configs, known issues, deployment process.

I listed my entire tech stack

Framework, database, hosting, UI library, APIs — everything.

Advanced CLAUDE.md Template

CLAUDE.md — advanced example 
# Project: My Awesome App
## Description
A client portal where customers can view their analytics,
manage subscriptions, and chat with support in real time.

## Tech Stack
- Frontend: React 18 + TypeScript + Tailwind CSS
- Backend: Supabase (auth + database + storage)
- Payments: Stripe
- Hosting: Vercel
- Email: SendGrid

## File Structure
src/
├── components/     → Reusable UI components
├── pages/          → Route-level page components
├── hooks/          → Custom React hooks
├── lib/            → Utilities, API clients, helpers
├── types/          → TypeScript interfaces
└── styles/         → Global styles and themes

## Coding Rules
1. ALWAYS use TypeScript — never plain .js files
2. Use shadcn/ui components for all new UI elements
3. All database queries go through Supabase client (src/lib/supabase.ts)
4. Never expose API keys in frontend code
5. Every component must have a clear, descriptive name
6. Use "const" by default, "let" only when reassignment is needed
7. Error handling: always wrap API calls in try/catch

## Things to NEVER Do
- Never modify the auth system without asking first
- Never delete migration files
- Never use "any" type in TypeScript
- Never commit .env files

## Deployment
- Production: my-app.vercel.app
- Preview: auto-deployed on every branch push
- Database: Supabase project "my-app-prod"

## Current Focus
Working on: subscription management page
Next up: admin analytics dashboard

I added coding rules and "things to never do"

The more specific your rules, the fewer mistakes Claude makes.

💡

Pro tip: Every time Claude makes a mistake, add a rule to prevent it next time. Your CLAUDE.md grows smarter over time. It's a living document.

Chapter 2

How Claude Remembers Things

The memory system explained simply
+100 XP
▼

Claude Code has 3 types of memory. Understanding them helps you work with Claude more effectively.

📄

1. CLAUDE.md (Manual)

YOU write this. Your rules, tech stack, project info. Claude reads it every session. It's the most important memory.

🗂️

2. Auto-Memory (Automatic)

Claude saves things it learns about you: your preferences, coding patterns, feedback you gave. Stored in a .claude/ folder.

💬

3. Conversation (Temporary)

The current chat session. Claude remembers everything said in this session. When you /clear, this resets.

How to tell Claude to remember things

messages you can type 
# Tell Claude to remember a preference
"Remember: I always want you to use Tailwind classes, never inline styles"

# Tell Claude to remember a project detail
"Remember: our API base URL is https://api.myapp.com/v2"

# Tell Claude to forget something
"Forget the rule about using semicolons — we removed that from our linter"

# Ask Claude what it remembers
"What do you remember about this project?"
"Check your memory for my coding preferences"

I understand the 3 types of memory

Manual (CLAUDE.md) + Auto-memory (.claude/) + Conversation (current session).

I told Claude to remember something

Try it: type "Remember: [something about your project]" in the chat.

🔄

Memory is cumulative. The more you work with Claude, the smarter it gets about YOUR project. After a few weeks, it will feel like working with someone who truly knows your codebase.

Chapter 4

Safety & Permission System

How to protect your code and secrets
+100 XP
▼
🛡️

Claude ALWAYS asks before doing anything risky. It won't delete files, push code, or change important configs without your explicit approval. You are always in control.

The 3 Permission Levels

🟢

Safe (Auto-allowed)

Reading files, searching code, analyzing. These happen automatically — no permission needed.

🟡

Moderate (Asks you)

Editing files, creating new files, running build commands. Claude shows you what it wants to do.

🔴

Dangerous (Always asks)

Deleting files, running destructive commands, pushing to GitHub. Always requires your YES.

Golden Safety Rules

Never put secrets in CLAUDE.md

API keys, passwords, tokens → .env file only. CLAUDE.md is for instructions, not secrets.

Always read Claude's changes before approving

When Claude says "I want to edit file X", read the diff. It takes 5 seconds and saves you hours.

Commit after every successful change

Use git commits as "save points". If something goes wrong, you can always go back.

My .gitignore includes .env

Open .gitignore and make sure ".env" is listed. This prevents accidentally sharing secrets.

⚠️

What if Claude does something wrong? Just press Ctrl + Z to undo, or type "undo that" in the chat. You can also close the file without saving. Nothing is permanent until you commit + push.

website

GitHub Copilot — The Autocomplete King

⚡

Copilot is best for: Real-time autocomplete while you type. It suggests the next line, completes functions, generates tests. Think of it as "predictive text for code".

setup steps
# 1. You need a GitHub account
https://github.com

# 2. Subscribe to Copilot ($10/month or free for students)
https://github.com/features/copilot

# 3. In VSCode: Extensions → Search "GitHub Copilot" → Install
# 4. Sign in with your GitHub account
# 5. Start typing code — suggestions appear in gray text
# 6. Press TAB to accept a suggestion

Can they coexist? YES! Use both at the same time.

Other AI Tools Worth Knowing

🔄

Continue (Free, Open Source)

Works with ANY AI model — Claude, GPT, Gemini, Llama, Mistral. You bring your own API key. Great for autocomplete + chat.

🤖

Cline / Roo Code

Autonomous agent that can plan and execute multi-step tasks. Like Claude Code but uses any model. Free extension, you pay for the API.

💻

Cursor IDE

A separate code editor (not VSCode) with AI built-in everywhere. Very powerful but $20/mo and you switch editors.

🌐

ChatGPT / GPT-4

Good for general questions, brainstorming, and explaining concepts. Use it in the browser. Not great for actual coding compared to specialized tools.

Mastery: Secrets of Power Users

+100 XP
▼

🎯 The 10 Rules of AI-Assisted Coding

Rule 1: Be specific, not vague

❌ "Fix the page" → ✅ "The /settings page returns a 500 error when saving — the crash is in updateProfile()"

Rule 2: One task at a time

Don't ask for 5 things at once. "Add a button" → approve → "Now style it" → approve → "Now add the logic".

Rule 3: Use /clear between different tasks

Switching from bug fixing to building a feature? Start fresh. Clean context = better results.

Rule 4: Commit after every success

Git commits are save points. If Claude breaks something later, you can always go back.

Rule 5: Give feedback

"Don't use that approach" or "I prefer it this way" teaches Claude. It remembers for next time.

Rule 6: Update CLAUDE.md regularly

New convention? Add it. New rule? Write it. Your CLAUDE.md should evolve with your project.

Rule 7: Use the right AI for the job

Claude for deep work, Copilot for autocomplete, Gemini for research. Don't use a hammer for screws.

Rule 8: Read the diff, always

Even if you trust the AI, spend 10 seconds reading what it changed. You'll catch issues early.

Rule 9: Ask "explain" when confused

After any change: "explain what you did and why". Claude will break it down simply.

Rule 10: Don't be afraid to say "no"

"No, undo that" or "Try a different approach". The AI works for you, not the other way around.
